Best practices

  • Here is a start on some basic best practices...

Start on a solid foundation

  • A good site install (Composer)
  • A good local environment setup
  • A solid, easy-to-work with hosting platform

Use full configuration exports

  • No reason to mess around with anything else
  • Start from day zero
  • IMO this SHOULD be the officially "recommended" workflow

Use version control

  • Without git or some kind of version control there is no good way to rollback or recover
  • Without git or some kind of version control there is no good way to rollback or recover

Start simple (stay simple)

  • Don't do things or use modules you don't actually need
  • e.g. config_split overboard
  • The system is designed to be SIMPLE

Develop and document a solid workflow

  • Your entire team needs to understand and buy in or the whole thing will fall apart

Be dependable and repeatable

  • If everything works different each time or breaks, you will lose buy in and cause frustration

Learn Drush

  • There are currently 8 config related Drush command to know and learn

Available commands

The big guns
  • drush config-export (cex)
  • drush config-import (cim)
  • drush config-delete
  • ...
The other guys
  • drush config-edit
  • drush config-get
  • drush config-list
  • drush config-pull
  • drush config-set
  • ...

Just say no -y

  • No chance to review and understand before taking action

Beware of history

  • There are an amazing amount of blog posts with old or no longer relevant tips, be careful

Develop an expert

  • You need to have at least one person on your team who understands and is driving this whole process
