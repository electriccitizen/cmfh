  • ...

Complications

  • Local only modules (Devel, Kint, UI)
  • Production only modules (SAML, Apache Solr)
  • Multiple people working on similar things
  • And often at the same time
  • Lost work and certain doom
  • ...

How to solve?

  • Everybody is still figuring this out but there are some common solutions

1. Contrib modules

  • There is a large and growing ecosystem of contrib modules to help fill in the gaps
  • The config split modules is one of the most common and widely used right now
  • It allows you to "split" or filter config per environment
  • This means you can have Devel or other modules running locally, but not on production
  • It is a pretty essential concept and some form of this will likely appear in core at some point

Other modules

There is a growing ecosystem of contributed config modules to solve common problems, e.g.
  • Config Ignore
  • Config Actions/Config Templates
  • Lots of others
  • Many of these have very few users right now
  • Some are pretty obscure
  • I will share a longer list later on

2. Git

Version control to the rescue
  • The whole point of putting config in code is that it can version controlled
  • You can roll back and save yourself when needed
  • In 2018 it simply doesn't make sense to manage your site with tarballs
